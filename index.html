HTTP Strict Transport Security (HSTS)

  • Chroni strony przed wymuszoną zmianą poziomu protokołu oraz przechwytywaniem sesji.
  • Powstał aby zapewnić korzystanie z protokołu szyfrowanego nawet w przypadku problemów konfiguracyjnych i błędów implementacji.
  • Przeglądarka sama podmieni adres na https
    • Przydatne np. w przypadku błędu z przekierowaniem na adres po http

HTTP Strict Transport Security (HSTS)

Rekomendowana jest podana niżej konfiguracja nagłówka ustawiająca go na dwa lata (max-age):

Strict-Transport-Security: max-age=63072000; includeSubDomains; preload

Nagłówek powinien zostać skonfigurowany dla głównej domeny, a domena powinna zostać dodana do listy domen HSTS Preload https://hstspreload.org/.

Jak to teraz jest z nagłówkiem X-XSS-Protection?

  • Krótkie wprowadzenie do ataków XSS.
  • Nagłówek miał na celu ochronę przed atakami Reflected XSS.
  • Określa w jaki sposób ma być wykorzystany wbudowany filtr pod kątem ataków Reflected XSS.
  • Działanie filtru opierało się na sprawdzeniu czy w treści przesyłanego żądania HTTP znajdują się tagi HTML lub fragmenty kodu JavaScript – oraz czy później wysyłane fragmenty kodu znajdują się w odpowiedzi.

Jak to teraz jest z nagłówkiem X-XSS-Protection?

X-XSS-Protection: 0
X-XSS-Protection: 1
X-XSS-Protection: 1; mode=block
X-XSS-Protection: 1; report=<reporting-uri>

Aktualne przeglądarki nie wspierają tego nagłówka!

https://caniuse.com/?search=x-xss-protection

Jak się zaczęła zła sława nagłówka X-XSS-Protection?

  • Nagłówek bezpieczeństwa wprowadzał podatność tam gdzie jej nie było!
  • Implementacja filtra polegała na zastąpieniu fragmentu uznanego za niebezpieczny skrypt znakiem #.
  • Błędy w implementacji powodowały że przez # zastępowany był fragment tagu HTML, payload nie był zmieniany, a modyfikacja innego elementu strony np. zamknięcia tagu style umożliwiała wykonanie ataku z wykorzystaniem składni CSS.

  • Prezentacja na temat podatności oraz https://markitzeroday.com/headers/content-security-policy/2018/02/10/x-xss-protection.html

Flaga HttpOnly

  • Większość aplikacji webowych wykorzystuje ciasteczka do utrzymania sesji użytkownika.
  • Dzięki zastosowaniu flagi JavaScript nie ma dostępu do wartości ciastka.
  • Prezentacja w aplikacji BadStore.

Flaga SameSite

  • Krótkie wprowadzenie do podatności CSRF, czyli zmuszenia przeglądarki do wykonania nieautoryzowanej akcji.
  • Flaga SameSite informuje przeglądarkę dla jakich żądań może ona dołączać ciastko.

CSRF, a flaga SameSite

 

  • Protokół HTTP jest bezstanowy - do identyfikacji użytkownika potrzebny jest mechanizm sesji.
  • Mechanizm ciastek.
  • Przeglądarka wyśle ciastko niezależnie od tego w jaki sposób doszło do „wyzwolenia” żądania.
  • Przykład można zobaczyć osadzając na innej stronie samowysyłający się formularz lub osadzając aplikację wewnątrz iframe <iframe src="text.pl"> lub nawet podając link do strony <a herf="test.pl">, czy odwołanie do zasobu <img src="test.pl">.

CSRF, a flaga SameSite

 

  • Strict powoduje blokowanie wysyłania  ciastek przez przeglądarkę we wszystkich kontekstach przeglądania z innych witryn, nawet jeśli wykorzystywany jest zwykły link do strony, czyli blokowane będą wszystkie żądania z innej domeny (żądania typu cross-site).
  • Lax pozwala przesłać ciastko po przejściu na stronę z zewnętrznego linku (nawigacja top-level), dozwolone są tzw. bezpieczne metody HTTP, czyli GET i HEAD (do tego grona należą także TRACE i OPTIONS) blokowane są natomiast metody wykorzystywane do ataków CSRF np. metoda POST.
  • None informuje przeglądarkę, że nie powinna blokować przesyłania ciastka w żądaniach cross-site
